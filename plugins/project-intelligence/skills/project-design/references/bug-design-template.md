# Bug 开发设计文档精简模板

Bug 文档默认贴近用户提供的样例，只保留定位和修复必需的信息。正文控制在 120 行以内；不要添加需求型文档的元数据、版本记录、源码证据表、接口/数据/UI 专章、测试矩阵或 AC 列表。

```markdown
# bug<编号> <Bug 名称>

## 1 BUG 现象

用一段话写清触发场景、当前错误行为和正确行为。

## 2 原因分析

列出直接根因。使用 `变量名`、`函数名` 或 `相对路径` 支撑结论，但不要单独展开源码证据表。

## 3 修复方案

### 3.1 改造思路

用 2 至 4 点说明实际修改方式和保持不变的逻辑。

### 3.2 新旧代码对照

仅展示与根因直接相关的最小旧代码和新代码。没有可靠旧代码时，改为简短的旧新逻辑对照，不编造代码。

### 3.3 逻辑变更说明

按主要业务分支说明修复后的行为。

## 4 影响范围

说明实际修改的文件或逻辑、受影响场景，以及明确不受影响的流程。

## 5 风险评估

**风险等级**：低 / 中 / 高

**极端预测**：最坏情况下会出现什么可观察问题。

**紧急举措**：如何回退或临时止损。
```

## 命名规则

- 纯数字 Bug 编号必须补上小写 `bug` 前缀：`1234` 生成 `bug1234-名称-设计文档.md`，标题使用 `# bug1234 名称`。
- 已以 `bug` 开头的编号保持原编号，不重复添加前缀。
- 非纯数字且没有 `bug` 前缀的正式编号保持原值；只有存在歧义时才询问。

## 精简规则

- 只把实际需要修改的文件写成改动点；用于理解调用链但不修改的文件，只在必要时作为一句背景说明。
- 默认不画 Mermaid 图，不列验收标准和测试矩阵。用户明确要求详细版时才增加。
- 代码对照应聚焦一个根因，避免复制完整函数或无关上下文。
- 影响范围和风险各保持一至三段，避免重复前文章节。
- 保持敏感信息脱敏，不保留模板占位符。
