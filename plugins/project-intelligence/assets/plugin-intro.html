Project-level Agent Intelligence

让 Claude Code 与 Codex 先懂项目，再写代码。

Project Intelligence 会把仓库里的组件、接口、服务、规范、图谱、质量检查和变更维护沉淀到 `.project-intel/`，让后续需求、debug、review 和重构都能从同一份项目事实出发。

Why It Exists

解决的不是“跑一个命令”，而是 Agent 不懂项目上下文。

很多代码助手每次都从零开始搜索文件。这个插件把项目共同点、硬规范、公共能力和图谱关系变成可复用事实源，让 Agent 的每次执行都有项目记忆。

Before

需求来了先翻代码

靠临时 grep 拼上下文，容易漏掉公共组件、已有 API、路由入口、权限边界和事务链路。

With PI

先查项目事实源

读取 `.project-intel/standards`、`knowledge`、`graph` 和 reports，再决定影响范围、复用候选和实现方式。

After

做完继续维护项目

执行后刷新规范与知识库，并按源码文件维护一份简短中文需求记录，减少后续追溯成本。

Lifecycle

从脑暴到维护，覆盖完整项目工作流。

不要求用户记住 CLI。日常可以直接说“帮我做这个需求”“帮我 review”“更新规范和知识库”，Agent 会映射到底层技能和检查脚本。

01

Brainstorm

澄清需求边界，比较实现路线，沉淀复用候选和关键风险。

02

Spec

按项目事实写需求说明、验收点、影响范围和约束。

03

Plan

把需求拆成可执行步骤，标记要读的规范、模块和测试。

04

Task

编码前查影响面和复用能力，避免重复造组件、Hook、服务和工具。

05

Review

结合 diff、规范、图谱、质量结果和测试缺口输出 findings。

06

Maintain

任务后 refresh/check/maintain，让项目知识库跟着代码更新。

Standards Extraction

不是空泛规范，而是从仓库事实里提炼规则。

初始化会提取前端和后端的公共能力、目录约定、重复模式、质量工具、图谱摘要和候选规范。规则按等级沉淀，不会把扫描猜测直接变成硬拦截。

Frontend

组件、Hooks、API、路由和状态

识别公共组件、页面局部组件、Props/Emits、请求封装、服务前缀、分包路由、Pinia store、样式变量和重复 UI 模式。

Backend

入口、服务、模型、Mapper 和边界

提取 Controller/API、Service、DTO/VO/Entity、Repository/Mapper、配置项、权限校验、事务、远程调用、消息任务和错误码。

Graph

GitNexus + Understand-Anything

GitNexus 负责符号调用和影响面；Understand-Anything 负责架构、模块、业务域和 onboarding 视角。插件只做统一摘要与编排。

前端细则

可沉淀的内容

  • `components.md`：公共组件、页面局部组件热点、重复组件候选。
  • `api.md`：请求封装、服务前缀、接口路径热点、API 模块清单。
  • `router.md`：分包、页面入口、自定义导航、插件 provider。
  • `domain-flows.md`：业务域候选、关键模块和图谱路径热点。
后端细则

可沉淀的内容

  • `backend-api.md`：框架入口、路径热点、非标准入口候选。
  • `backend-services.md`：服务方法、事务信号、远程调用、权限信号。
  • `backend-security.md`、`backend-transactions.md`、`backend-errors.md`：安全、事务和错误码边界。
  • `backend-repository.md`、`backend-config.md`、`backend-async.md`：数据访问、配置、消息和定时任务。
hard

结构化规则自动门禁，纯文本规则进入 manual-review，不误判通过。

preferred

项目中高频稳定模式，作为默认建议和 review 依据。

inferred

扫描推断出来的规则，需要团队后续确认。

candidate

重复出现但未封装的沉淀候选，只提示，不默认阻塞。

Natural Language First

用户说人话，Agent 走项目流程。

CLI 是底层工具。更推荐通过对话触发：Agent 会读 `.project-intel`、图谱和规范；spec/plan 默认只在上下文整理，明确要求才写文件。

初始化这个项目规范 执行 `init`，生成 `.project-intel`、前后端标准、知识库、图谱摘要、质量报告和本地 agent 入口。
帮我做这个需求 进入 `project-task`：先整理轻量中文 spec，查影响面和复用候选，再实现，最后 check + maintain。
帮我查询这个 bug 进入 `project-debug`：先复现和定位根因，读取规范/图谱/最近变更，再提出修复。
帮我 review 代码 读取 diff、硬规范、质量检查、GitNexus 影响面、Understand-Anything 架构上下文，按严重度输出 findings。
帮我更新规范和知识库 执行无安装副作用的 `refresh`，扫描当前工作区和别人 pull 下来的提交；需要重跑图谱时显式使用 `refresh --with-graph`。
Repository Fact Layer

所有长期事实都落在 `.project-intel/`。

规范和轻量知识库可以提交进仓库；缓存、中间产物和大型原始图谱默认不提交，保持仓库内事实层轻量可审查。

.project-intel/manifest.json

技术栈、扫描时间、图谱来源、版本、Git commit 和刷新状态。

.project-intel/standards/*.md

前端、后端、质量、复用、API、组件、事务、权限、错误码等规范。

.project-intel/knowledge/*.json

结构化组件、Hooks、API、服务、DTO/VO、Entity、Mapper、路由和配置。

.project-intel/graph/project-graph.json

统一引用 GitNexus 和 Understand-Anything 的轻量图谱摘要。

.project-intel/reports/*.md

初始化、质量、冗余、任务影响、debug context 和 review 报告。

.project-intel/requirements/files/**/*.md

按源码文件维护一份简短中文需求记录，避免每轮对话生成大量散文件。

Install

Claude Code 和 Codex 都能使用。

推荐通过 npm 同时安装 CLI 与双端插件包；注册 Agent 是显式动作，不会在安装依赖时静默改写 Claude Code 或 Codex 配置。

npm 推荐

CLI + 双端插件包

需要 Node.js 18+ 与 Python 3.9+：

npm install -g project-intelligence
project-intel agent install --target all
project-intel init

全局 CLI、skills 和两套 marketplace 会使用同一版本；安装后启动新的 Agent 会话。

Claude Code

Slash UI 安装

在 Claude Code 对话里执行：

/plugin marketplace add crayonYYxm/project-intelligence
/plugin install project-intelligence@project-intelligence

安装后启动新的 Claude Code 会话，让 plugin skills 生效。

如果初始化时要启用 Understand-Anything，Claude Code 侧会提示安装/启用后运行 `/reload-plugins`，再运行 `/understand . --language zh`，完成后执行 `/project-refresh`。
Codex

Marketplace 安装

在终端执行：

codex plugin marketplace add crayonYYxm/project-intelligence
codex plugin add project-intelligence@project-intelligence

安装后启动新的 Codex 线程加载插件 skills。

Codex 侧安装 Understand-Anything 增强图谱时，插件会使用 `install.sh codex`；GitNexus 通常通过 `npx gitnexus analyze` 准备并分析。
初始化

推荐第一步

project-intel init

会检查 GitNexus、Understand-Anything、Node/package manager 和质量工具；非交互环境只给建议，`--interactive` 才询问，`--setup-missing` 才安装。

图谱增强

GitNexus + Understand-Anything

GitNexus 提供符号级影响分析，Understand-Anything 提供架构和领域视角；两者都是可选增强，缺失时仍会生成基础项目索引。

Agent Contract

让每次代码修改都带着项目上下文。

Project Intelligence 的核心不是多生成几个文件，而是把“先理解项目，再执行需求，做完继续维护”写进 Agent 工作方式。
