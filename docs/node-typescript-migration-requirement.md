# LOCAL-20260721-144445 Project Intelligence Node.js/TypeScript 运行时迁移 需求文档

## 1. 文档信息

- 需求号：LOCAL-20260721-144445
- 需求名称：Project Intelligence Node.js/TypeScript 运行时迁移
- 单据类型：Requirement
- 文档状态：可评审
- 当前基线：project-intelligence 0.6.1
- 目标运行时：Node.js 18 及以上版本

## 2. 背景与现状

Project Intelligence 当前以 npm 包形式交付，用户通过 `project-intel` 或 `project-intelligence` 命令调用产品。npm 的可执行入口由 Node.js 启动，但核心命令、项目扫描、需求生命周期、测试证据、评审收口、适配器和文档处理仍由 Python 实现，Node.js 入口实际会继续拉起 Python 进程。

该结构在功能交付上存在以下问题：

1. npm 无法完整表达实际运行依赖。用户看到的是 Node.js 包，但运行时仍必须另外安装 Python 3.9 及以上版本。
2. 跨平台行为由 Node.js、Python、操作系统 Shell 和字符编码共同决定。Windows 已出现非 UTF-8 控制台导致命令失败的情况，Git Hook 生成逻辑也默认使用 `python3`，与 Windows 的常见运行环境不一致。
3. 安装、诊断、测试和发布需要同时维护两套运行时，增加了版本漂移、环境探测和故障定位成本。
4. 当前主要实现约为 1.2 万行 Python 代码，不能采用一次性重写方式切换，否则容易遗漏现有 CLI、文件格式和生命周期规则。

本需求采用渐进式等价迁移。迁移完成后，面向用户发布的 npm 包仅依赖 Node.js；TypeScript 作为主要开发语言，发布时编译为 JavaScript，不要求用户安装 TypeScript 或 Python。

## 3. 目标

1. 将 Project Intelligence 的生产运行时从“Node.js 启动器 + Python 核心”迁移为“Node.js + TypeScript 核心”。
2. 最终发布包在未安装 Python 的环境中可完成安装、初始化、诊断、项目分析和完整需求生命周期操作。
3. 保持现有公开 CLI 的命令名称、参数、帮助信息、退出码和 JSON 输出协议兼容。
4. 保持 `.project-intel` 的目录结构、配置、需求文档、状态记录、测试证据和生成物格式兼容，已有项目无需人工转换即可继续使用。
5. 保持 GitNexus、Understand-Anything、Git Hook、Skill 和插件清单等现有集成能力不变。
6. 在 Windows、macOS 和 Linux 上形成一致的 UTF-8、路径、子进程和文件写入行为。
7. 建立可重复执行的双实现对照测试和迁移验收证据，确保功能迁移过程中不静默丢失既有行为。

## 4. 业务场景

### 4.1 新用户安装

用户的设备仅安装符合版本要求的 Node.js。用户通过 npm 全局安装 Project Intelligence 后，可以直接执行版本查询、诊断、初始化和项目命令，不需要安装或配置 Python。

### 4.2 已有项目升级

用户将 0.6.1 或更早兼容版本升级到 Node.js/TypeScript 实现后，现有 `.project-intel` 数据可被直接读取和更新，需求状态、测试证据、图谱引用和生成文档不丢失，也不要求用户执行单独的数据迁移命令。

### 4.3 跨平台协作

同一个项目可由 Windows、macOS 和 Linux 用户共同维护。不同平台执行同一命令时，除绝对路径和平台固有信息外，结构化结果、文件内容、排序规则、错误类型和退出码保持一致。

### 4.4 需求全生命周期

用户可继续执行初始化、需求登记、规格化、计划、测试、评审、完成和维护等流程。迁移只替换实现技术，不改变生命周期状态机、门禁条件、证据要求和 Skill 使用方式。

### 4.5 工具与仓库集成

用户可继续安装和使用 Git Hook、适配器、GitNexus、Understand-Anything 及插件资源。由产品生成的 Hook 应调用当前 npm 包提供的 Node.js CLI，不依赖系统中的 `python3` 命令。

### 4.6 开发与发布

维护者使用 Node.js 完成依赖安装、类型检查、单元测试、跨平台测试、构建、打包和发布。npm 包中包含可直接运行的 JavaScript 产物及必要资源，并能从包版本生成唯一、准确的版本输出。

## 5. 范围

### 5.1 功能范围

本次迁移覆盖以下现有能力：

| 能力域 | 迁移内容 | 兼容要求 |
| --- | --- | --- |
| CLI 与应用入口 | 命令解析、全局选项、帮助、版本、文本与 JSON 输出 | 命令、参数、退出码和输出协议保持兼容 |
| 项目初始化与诊断 | `init`、`doctor`、`check`、安装与环境探测 | Node-only 环境可运行，诊断信息准确 |
| 项目事实层 | refresh、扫描器、框架识别、包管理器与质量工具识别 | 固定样例的归一化扫描结果等价 |
| 需求生命周期 | intake、spec、plan、requirement、lifecycle、debug、test、review、finish、maintain | 状态机、门禁、文档和证据规则保持兼容 |
| 查询与图谱 | requirements、query、graph-tools 及图谱来源状态 | 参数、结果结构和降级行为保持兼容 |
| 适配与自动化 | adapters、agent install、Git Hook 生成与卸载 | 生成物改用 Node.js CLI，行为保持兼容 |
| 文件与文档 | YAML、JSON、Markdown、哈希、时间戳、锁、原子写入 | 已有数据可直接读写，不产生破坏性变更 |
| 外部进程 | Git、npm、GitNexus、Understand-Anything 及质量命令调用 | 超时、信号、标准流和错误映射一致 |
| 工程化 | TypeScript 编译、测试、CI、npm 打包、发布校验 | 发布包不需要 Python，三平台自动验收 |

### 5.2 迁移阶段

1. 建立行为基线：冻结公开 CLI 清单、输出协议、文件样例和现有测试映射，新增 Python 与 Node.js 的对照测试工具。
2. 迁移基础设施：实现 TypeScript CLI、配置、文件系统、路径、进程、输出、锁和文档基础能力；尚未迁移的能力在开发阶段可通过兼容桥接调用旧实现。
3. 迁移业务模块：按项目扫描、需求生命周期、测试证据、评审收口、适配器和图谱集成的顺序迁移，并逐模块切换到 Node.js 实现。
4. 移除生产桥接：所有公开命令完成等价验收后，删除发布包对 Python 进程和 Python 模块的运行时调用。
5. 发布验证：通过干净环境安装、三平台兼容、已有项目升级和回滚读取验证后，更新安装文档并发布正式版本。

开发分支和 CI 在迁移期间可以保留 Python 旧实现用于对照验证；正式发布包达到本需求验收条件时不得再以 Python 作为运行依赖或功能回退路径。

### 5.3 代码与包结构

1. 新核心代码使用 TypeScript，并在发布前编译为 Node.js 可执行的 JavaScript。
2. CLI 入口保持 `project-intel` 和 `project-intelligence` 两个命令别名。
3. 运行时资源通过 npm 包统一定位，不依赖仓库工作目录或开发机绝对路径。
4. Python 测试应逐项映射为 Node.js 测试，允许重组测试文件，但不得无说明删除所覆盖的行为。
5. Python 源码可在迁移期保留为对照基线；移除时间以生产运行路径完全不再引用 Python 为准。

## 6. 非目标

1. 本需求不新增业务命令、生命周期阶段或项目事实类型。
2. 本需求不重新设计 `.project-intel` 数据模型、需求模板、Skill 内容或插件目录规范。
3. 本需求不改变 GitNexus 和 Understand-Anything 的产品边界、安装方式或输出语义。
4. 本需求不进行站点、图形界面或交互视觉改版。
5. 本需求不把性能优化作为独立目标；迁移过程中仅要求避免明显性能回退和资源泄漏。
6. 本需求不处理独立的安全加固项。迁移实现仍需保持现有安全边界，不得因语言迁移引入新的功能性回归。
7. 未在 README、CLI 帮助或发布文档中承诺的 Python 模块直接导入方式，不作为长期公开 API 保留；如实施前发现已有正式文档承诺，应提供至少一个次版本的兼容或弃用说明。

## 7. 业务规则与异常边界

1. **运行时规则**：正式发布版本仅要求 Node.js 18 及以上版本。TypeScript 编译器属于开发依赖，不属于用户运行依赖。
2. **CLI 规则**：公开命令、参数名称、默认值、互斥关系、帮助信息、文本输出意图、JSON 字段、状态值和退出码均为兼容合同。
3. **数据规则**：已有 `.project-intel` 文件必须原地可读。未获得独立的数据模型变更批准时，不得提高 schema 版本或写入旧版本无法安全忽略的字段。
4. **编码规则**：命令标准输出、标准错误、生成的 JSON、YAML 和 Markdown 默认使用 UTF-8；中文项目名、需求名和错误信息在三平台均可正常处理。
5. **路径规则**：相对路径以显式项目根目录为基准；Windows 盘符、UNC 路径、POSIX 绝对路径、空格和非 ASCII 字符均须通过固定样例测试。
6. **文件规则**：锁、临时文件、原子替换、目录创建、权限错误和并发写入不得造成半写文件或静默覆盖。失败后应保留可诊断错误和原始有效数据。
7. **结果稳定性规则**：目录扫描、列表和 JSON 数组的排序必须确定；哈希输入、换行归一化和时间字段的生成规则必须有测试约束。
8. **子进程规则**：外部工具调用应明确参数数组、工作目录、环境变量、超时、退出码、信号和输出编码。工具不存在、超时或返回非零状态时，应映射为与现有 CLI 等价的可诊断结果。
9. **兼容桥接规则**：迁移期间允许内部桥接旧实现，但每个桥接点必须可被测试识别并有对应移除条件；正式发布验收会扫描 Python 进程调用和运行时文件引用。
10. **回滚规则**：Node.js 实现写入的数据不得阻止当前基线版本读取同一 schema。升级或执行命令失败时，不得破坏用户回滚 npm 版本后继续使用项目数据的能力。
11. **版本规则**：`--version`、`doctor`、npm 元数据和发布标签必须来自同一版本源，不得分别硬编码。
12. **异常规则**：参数错误、项目未初始化、配置损坏、外部工具缺失、文件无权限和业务门禁未满足必须保持可区分的错误类型，并在 JSON 模式中返回稳定结构。

## 8. 验收标准

- AC-01：在仅安装 Node.js 18 及以上版本、PATH 中不存在 Python 的干净环境中，npm 安装成功，且 `project-intel --version`、`project-intel doctor --json` 和 `project-intel init --dry-run --no-graph` 均可正常执行。
- AC-02：以 0.6.1 为基线生成公开 CLI 快照，Node.js 实现的命令、子命令、全局参数、子命令参数、默认值、互斥关系和帮助信息全部通过兼容性测试；任何有意差异均须另立需求批准。
- AC-03：成功、用法错误、项目状态错误、外部工具错误和内部运行错误五类固定场景的退出码、JSON 顶层字段、状态值和错误分类与基线兼容。
- AC-04：至少使用包含初始化配置、项目事实、需求生命周期、测试证据、评审结果和维护记录的 0.6.1 数据夹具验证升级；Node.js 实现无需人工转换即可读取、查询和更新，且更新前后没有未授权的数据丢失或 schema 变化。
- AC-05：`init → intake → spec → requirement ready → begin → test → review → finish → maintain` 主流程及诊断、延期、重开、修订和发现项处理分支均有端到端测试，并保持现有状态门禁和失败原因。
- AC-06：针对 Python、JavaScript/TypeScript、Java 和混合仓库固定夹具，项目扫描、框架识别、包识别、质量命令识别和归一化事实输出达到字段级等价；时间、临时路径等非确定字段在比较前按书面规则归一化。
- AC-07：adapters 和 agent install 的预览、应用、状态和移除流程通过测试；生成的 Git Hook 调用 npm 包提供的 Node.js CLI，在 Windows、macOS 和 Linux 上均不依赖 `python` 或 `python3`。
- AC-08：Windows、macOS 和 Linux CI 全部通过；每个平台均覆盖中文项目路径、中文需求内容、带空格路径、长输出和非 UTF-8 控制台环境下的 UTF-8 结果验证。
- AC-09：对 `npm pack` 产物执行全新临时目录安装和核心流程冒烟测试；测试期间监测子进程，确认没有查找、启动或动态导入 Python，且运行时不读取 npm 包之外的仓库源码。
- AC-10：现有 Python 自动化测试逐项建立“原测试 → Node.js 测试或等价端到端场景”的映射清单，所有当前有效测试均有对应覆盖；未获批准的测试删除、跳过或弱化会导致验收失败。
- AC-11：故意制造真实测试失败时，test、review 和 finish 门禁必须拒绝通过；仅包含伪造通过文本但没有受支持测试报告或可验证命令结果的证据不得被判定为成功。
- AC-12：`project-intel --version`、`project-intel doctor --json`、`package.json` 和打包产物报告同一版本；发布校验能够在版本不一致时失败。
- AC-13：由 Node.js 实现更新后的兼容 schema 数据，可由 0.6.1 基线读取；执行失败和版本回滚场景不会留下损坏的 JSON、YAML、Markdown 或锁文件。
- AC-14：最终 npm 包的安装文档不再要求 Python，运行依赖清单不包含 Python，生产入口及生成的 Hook 不引用 Python，仓库发布检查可自动阻止上述依赖重新出现。
- AC-15：在固定中型仓库上记录 0.6.1 与 Node.js 实现的启动、初始化预览和刷新基准；Node.js 实现不存在无限等待、僵尸进程或持续增长的句柄，三个场景的中位耗时均不高于基线的 1.25 倍。

## 9. 外部接口影响

不影响对外接口（HTTP/API）：项目当前不以 HTTP 服务接口作为本次迁移的交付面，本需求不新增、删除或修改对外网络 API。

本需求影响本地公开 CLI 的内部实现和运行依赖，但不得造成破坏性接口变化。以下内容按外部兼容合同处理：

1. `project-intel` 与 `project-intelligence` 命令名。
2. 所有已发布子命令、选项、帮助语义和退出码。
3. `--json` 输出结构及机器可读状态值。
4. `.project-intel` 中已发布的配置、状态、文档和证据格式。
5. Git Hook、Skill、GitNexus、Understand-Anything 和自动化脚本对 CLI 的调用方式。
